2. Linux raw socket receive semantiği (kaynaklı)  
AF_INET/SOCK_RAW/IPPROTO_TCP ile recvfrom(2) IP header + TCP header döner (raw(7): “For receiving, the IP header is always included in the packet.”). L2 frame asla teslim edilmez (raw(7) ve packet(7) karşılaştırması: AF_PACKET L2, AF_INET L3).  
SO_ATTACH_FILTER’in gördüğü buffer ile recvfrom buffer aynıdır: sk_buff→data IP header başlangıcında (offset 0 = version/IHL); BPF VM bu buffer üzerinde çalışır (socket(7) ve kernel include/uapi/linux/filter.h: BPF_LD ABS/IND).  
bind(local_ip, 0) receive filtering sağlar: “A raw socket can be bound to a specific local address… If it isn’t bound, all packets with the specified IP protocol are received” (raw(7)). Port 0 ile bind scope etmez, sadece local IP’yi filtreler (inet_bind + raw_bind path, net/ipv4/raw.c).  
connect(2) receive filtering sağlamaz; sadece sendto için route cache yaratır.  
SO_BINDTODEVICE receive’i interface’e bağlar (socket(7)).  
ip_early_demux=1 (varsayılan) TCP SYN-ACK’ı raw socket’e ulaşmadan consume etmez; raw_rcv() ip_local_deliver_rcu’da protocol handler’lardan önce çağrılır (raw(7): “passed to any raw sockets … before it is passed to other protocol handlers”). Kernel path: ip_rcv → ip_rcv_finish → ip_local_deliver → ip_local_deliver_finish → raw_local_deliver (clone) → tcp_v4_rcv.  
rp_filter raw paketleri etkiler (ip_rcv’de strict/ loose reverse path check).  
IPPROTO_TCP ile raw socket kernel’de ip_protocol→handler = raw_rcv ile demux olur (net/ipv4/af_inet.c: inet_add_protocol).  

3. Tüm makul senaryolar  
S1–S16 tam listesi sorguda verildiği gibi ele alındı (byte order, MSH offset, ip_early_demux, rp_filter, conntrack INVALID, BPF_RET, TX loopback, netfilter INPUT drop, docker namespace vs.).  

4. Senaryoların tek tek elenmesi  
S1 (byte order): Elimizdeki kanıtla uyuşmuyor → önceki bind+userspace filter ile diğer TCP paketleri düşüyordu; filter sadece bu flow’da sorunlu olsaydı diğer trafik de etkilenirdi.  
S2 (offset mantığı): Uyuşmuyor → LD W ABS 16 incoming dst IP’dir (IP header offset 16), LDX MSH + LD H IND doğru TCP port offset’i üretir.  
S3 (0xb1 MSH desteği): Reddedildi → 6.19 kernel’de LDX|B|MSH tam destekli (uapi/filter.h).  
S4 (sk_buff vs tcpdump buffer): Reddedildi → tcpdump AF_PACKET L2, raw AF_INET L3 ama IP header aynı; SYN-ACK hex uyumlu.  
S5 (ip_early_demux consume): Eleniyor → raw(7) ve kernel path: raw_rcv protocol handlers’dan önce.  
S6 (rp_filter drop): Eleniyor → tcpdump SYN-ACK’i görüyor, rp_filter ip_rcv’de erken drop yapardı.  
S7 (conntrack INVALID): Confirmed → OUTPUT NOTRACK var, INPUT/PREROUTING NOTRACK yok → SYN-ACK ctstate=INVALID → INPUT chain’de (varsa) drop veya raw delivery öncesi tüketim (netfilter LOCAL_IN hook ip_local_deliver_finish öncesi).  
S8 (TX loopback): Eleniyor → sendto raw socket’e loopback yapmaz (no IP_MULTICAST_LOOP, no local delivery for self-generated outgoing).  
S9 (SO_RCVTIMEO race): Eleniyor → strace recvfrom sendto’dan sonra başlıyor.  
S10 (bind eksik): Eleniyor → tarihsel bind(local_ip,0) çalışıyor.  
S11 (protocol demux): Eleniyor → raw(7) “passed to both the kernel module and the raw socket(s)”.  
S12 (INPUT iptables drop): S7 ile örtüşüyor, muhtemel yardımcı.  
S13 (BPF_RET 0x00040000): Reddedildi → standart socket filter’da 0x40000 = accept (packet length >= return → full pass, socket(7) ve libpcap örnekleri).  
S14 (docker): Eleniyor → docker0 DOWN, DOCKER chain’ler aktif olsa bile enp37s0 traffic’i etkilemez.  
S15 (multiple raw socket delivery): Eleniyor → tek socket.  
S16 (tcpdump vs raw guarantee): Uyuşmuyor → tcpdump görüyorsa paket ip_rcv’e girdi; raw drop netfilter LOCAL_IN’de olur.  

5. 11-instruction cBPF filter’ın adli analizi  
5a. Instruction-by-instruction çözümleme  
ins[0] 0x30 LD B ABS 9 → IP proto (offset 9).  
ins[1] 0x15 JEQ k=IPPROTO_TCP jt=0 jf=8 → TCP değilse drop.  
ins[2] 0x20 LD W ABS 16 → incoming dst IP (offset 16).  
ins[3] 0x15 JEQ k=src_ip (local 192.168.1.2 network order) jt=0 jf=6.  
ins[4] 0xb1 LDX B MSH 0 → X = (byte[0] & 0xf) * 4 = IHL*4.  
ins[5] 0x48 LD H IND 0 → TCP sport (X+0).  
ins[6] 0x15 JEQ k=target_port (htons(443)) jt=0 jf=3.  
ins[7] 0x48 LD H IND 2 → TCP dport (X+2).  
ins[8] 0x15 JEQ k=ephemeral_port (htons(62893)) jt=0 jf=1.  
ins[9] 0x06 RET 0x00040000 → ACCEPT (full packet).  
ins[10] 0x06 RET 0 → DROP.  
5b. Gerçek SYN-ACK paketi üzerinde elle yürütme  
SYN-ACK IP header: version/IHL=0x45, proto=0x06, dst=0xc0a80102, sport=0x01bb, dport=0xf5ad (62893 network).  
ins[0]→6, ins[1]→TCP, ins[2]→0xc0a80102 (eşleşir), ins[4] X=20, ins[5]→0x01bb (eşleşir), ins[7]→0xf5ad (eşleşir) → ins[9] ACCEPT. Filter doğru çalışır (byte order doğru varsayımıyla).  
5c. Byte order analizi  
k alanları packet endian’ı ile aynı olmalı (network byte order). src_ip = htonl(0xc0a80102), portlar = htons(). Zig’de @byteSwap veya posix.htons kullanılmadıysa S1 confirmed olurdu ama tarihsel trafik çalıştığı için büyük ihtimalle doğru.  
5d. BPF_RET doğruluğu  
0x00040000 klasik socket filter’da ACCEPT’tir (socket(7): “nonzero value … >= packet’s data length → unmodified”; libpcap ve kernel örneklerinde standart).  

6. Crafted SYN paket analizi  
IP: v4, IHL=5, len=0x003c, TTL=0x40, proto=6, src=192.168.1.2, dst=1.1.1.1, checksum doğru.  
TCP: sport=0xed1b (tcpdump ile uyumsuzluk not: hex farklı run’a ait olabilir), dport=0x01bb, SYN flag (0xa002), seq=0x02878109, window=0xffff, options (MSS 1460, SACK, TS, WS 7) doğru format.  
IP_HDRINCL set ise kernel checksum, src (zero ise), ID, len doldurur (raw(7) tablosu). Checksum’lar wire’da doğru (tcpdump SYN-ACK aldı). SYN’deki kusurlar elendi.  

7. TX/RX aynı fd mimarisi analizi  
raw(7): aynı fd üzerinde sendto+recvfrom desteklenir ve güvenilirdir. TX paketi loopback etmez (no multicast loop semantics). Ayrı RX fd açmak delivery path’ini değiştirmez. AF_PACKET RX + AF_INET TX kombinezonu mümkün ancak AF_INET yeterli.  

8. Kernel delivery path analizi  
8a. ip_early_demux etkisi: raw_rcv önce çağrılır, consume etmez (net/ipv4/ip_input.c).  
8b. conntrack/NOTRACK etkileşimi: OUTPUT NOTRACK SYN’i ctstate yaratmaz → SYN-ACK PREROUTING/INPUT’ta INVALID → netfilter LOCAL_IN’de drop (nf_conntrack_tcp_packet → ctstate=INVALID). Raw delivery ip_local_deliver_finish sonrası.  
8c. rp_filter etkisi: ip_rcv’de, tcpdump görüyorsa drop yok.  
8d. tcpdump vs raw socket: tcpdump AF_PACKET (netif_receive_skb öncesi), raw ip_local_deliver sonrası; netfilter drop ikisi arasında olabilir.

10. Minimal fix veya mimari karar  
Bu yaklaşım (AF_INET/SOCK_RAW TX+RX) Linux’ta reliable değildir çünkü conntrack/netfilter LOCAL_IN raw_rcv’den önce çalışır. ip_early_demux veya conntrack application-level fix ile çözülemez (kernel path zorunlu).  
Minimal patch: raw PREROUTING’e NOTRACK ekle (S7’yi çözer). AF_PACKET/SOCK_RAW (ETH_P_IP) ile RX’e geçmek daha robust (packet(7)).

Çekirdek ve Sistem Mimarisi
Sistem, 6.19.10-1-cachyos-bore-lto çekirdek versiyonu ile çalışan, performansa yönelik optimize edilmiş bir Arch Linux türevidir. Bu modern çekirdek, gelişmiş ağ yığını optimizasyonlarını (örneğin güncel ip_early_demux, nf_conntrack mekanizmaları ve eBPF/cBPF JIT derleyicisi) barındırmaktadır. verify.sh içindeki bilgilendirme mesajından ve işlemlerden anlaşıldığı üzere, hedef uygulamanın (ghost_engine) raw soket açabilmesi için elzem olan CAP_NET_RAW yetkisi eksiksizdir.B. Soket ve Filtre KonfigürasyonuHedef uygulama, AF_INET, SOCK_RAW, IPPROTO_TCP parametreleriyle oluşturulmuş bir dosya tanımlayıcısı (file descriptor) kullanmaktadır. strace kayıtlarına göre, sisteme iletilen cBPF (Classic Berkeley Packet Filter) yönergeleri, ilk EFAULT belleklendirme hatasından sonra düzeltilmiş ve setsockopt(..., SO_ATTACH_FILTER,...) sistem çağrısı 0 (başarılı) dönüş kodu ile sonuçlanmıştır. Çekirdeğin filtreyi doğrulaması (verification) başarıyla tamamlanmıştır.C. Asimetrik Netfilter (Iptables) KurallarıSisteme enjekte edilen güvenlik duvarı kuralları incelendiğinde, büyük bir asimetri saptanmıştır:iptables -t raw -A OUTPUT -p tcp --sport {port} -j NOTRACK
Bu kural, kaynak port üzerinden dışarı çıkan (TX) TCP paketlerinin nf_conntrack durum tablosuna (state table) kaydedilmesini engellemektedir. Ancak, elde edilen verilerde, INPUT (veya PREROUTING) zincirinde aynı paketin dönüş yolunu (SYN-ACK) izleme dışı bırakacak simetrik bir NOTRACK kuralının bulunmadığı tespit edilmiştir.D. L2/L3 İletimi ve DoğrulamaAğ dinleme aracından (tcpdump) alınan çıktı, 192.168.1.2:62893 adresinden 1.1.1.1:443 adresine gönderilen SYN paketinin, tüm Checksum ve IP kurallarına uygun olarak fiziksel ağ arayüzünden dışarı çıktığını kanıtlamaktadır. Daha da önemlisi, 1.1.1.1:443 adresinden 192.168.1.2:62893 hedefine doğru SYN-ACK paketi arayüze fiziksel olarak geri dönmüş ve tcpdump tarafından yakalanmıştır. Bu veri, ağ rotalamasının (routing), NAT cihazlarının ve İnternet Servis Sağlayıcı katmanlarının sorunsuz çalıştığını tartışmasız bir şekilde ispatlar.E. İletim Arızası (Delivery Failure)Fiziksel olarak enp37s0 arayüzüne ulaşan ve tcpdump tarafından görülen SYN-ACK paketi, hedef uygulamanın recvfrom() sistem çağrısına teslim edilmemekte ve süreç EAGAIN ile 5 saniyelik zaman aşımına (timeout) uğramaktadır. Paket, donanım sürücüsü ile kullanıcı alanı soket kuyruğu (userspace socket receive queue) arasındaki işletim sistemi ağ yığınında (kernel network stack) düşürülmekte (dropped) veya filtrelenmektedir.2. Linux Raw Socket Receive SemantiğiLinux çekirdeğinde AF_INET alanında, SOCK_RAW tipi ve spesifik bir protokol numarası ile açılan soketlerin alım (receive) semantiği, geleneksel iletim denetimi soketlerinden köklü biçimde farklıdır ve sıkça yanlış yorumlanan çekirdek davranışları barındırır.2a. Buffer İçeriği ve SınırlarıAF_INET / SOCK_RAW soketleri üzerinden recvfrom() çağrısı ile alınan tampon bellek (buffer), her zaman ağ katmanı başlığını (IPv4 Header) ve ardından gelen iletim katmanı başlığı ile veri yükünü (TCP Header + Payload) içerecek şekilde kullanıcı alanına teslim edilir. L2 (Ethernet/MAC) çerçevesi, ağ kartı sürücüsü ve çekirdekteki __netif_receive_skb_core() fonksiyonu tarafından soyutlandığı için AF_INET soketlerine hiçbir zaman iletilmez. Dolayısıyla, sk_buff yapısındaki veri işaretçisi (skb->data), paket raw_v4_input döngüsüne ulaştığında doğrudan IPv4 başlığının ilk baytını gösterir.2b. SO_ATTACH_FILTER ve sk_buff Görüş AçısıSokete SO_ATTACH_FILTER ile iliştirilen cBPF filtresi, sock_queue_rcv_skb() aşamasında, paket soketin kuyruğuna alınmadan hemen önce sk_filter() çekirdek fonksiyonu aracılığıyla çalıştırılır. AF_INET tabanlı raw soketler için BPF sanal makinesindeki (VM) 0 ofseti, recvfrom() fonksiyonunun gördüğü alanla tam olarak aynıdır: IP başlığının ilk baytı.2c. Bind Çağrısı ve Port KapsamıÇekirdek kaynak kodu incelendiğinde (net/ipv4/raw.c içerisindeki raw_bind() ve raw_v4_input() fonksiyonları), SOCK_RAW için yapılan bind() çağrısının yalnızca yerel IP adresini (sin_addr) bağladığı açıkça görülmektedir. Çekirdek, sin_port alanını tamamen göz ardı eder. Bu nedenle bind(local_ip, 0) çağrısı, raw soketi belirli bir TCP portuna kısıtlamaz. Raw soket, port bağımsız olarak, belirtilen protokole (IPPROTO_TCP) sahip olan ve IP adresi eşleşen tüm paketlerin potansiyel alıcısıdır.2d. Connect Çağrısı ve FiltrelemeRaw soket üzerinde connect() çağrısı yapılması, ağ yığınında gerçek bir oturum kurmaz; yalnızca soketin yerel durum yapısındaki uzak hedef IP adresini (sk->sk_daddr) kaydeder. Bu işlem, çekirdeğin raw_v4_input() sırasında paketleri filtrelerken hedeften gelmeyen paketleri erken aşamada düşürmesini sağlayan bir "teslimat filtresi" görevi görür. Ancak bu, sorunun doğasını değiştirmez.2e. SO_BINDTODEVICE EtkisiSO_BINDTODEVICE seçeneği, soketin belirli bir fiziksel arayüze (örneğin enp37s0) kilitlenmesini sağlar. Bu kilitlenme, çekirdekte sk_buff içerisindeki skb->dev->ifindex ile soketin tanımlayıcısı arasındaki basit bir eşleştirme kontrolünden ibarettir. Paket zaten doğru arayüzden geldiği için bu konfigürasyon, incelenen arıza durumunun bir parçası değildir.2f. ip_early_demux EtkileşimiLinux ağ yığınında paket işleme optimizasyonu sağlayan ip_early_demux (Linux 3.6 ve sonrasında varsayılan olarak 1), paketlerin yönlendirme (routing) tablosu maliyetini azaltmak için paketleri önceden kurulu TCP/UDP soketleriyle eşleştirir. Ancak net/ipv4/ip_input.c içerisindeki ip_local_deliver_finish() fonksiyonunda, raw_local_deliver() çağrısı her zaman ip_early_demux tarafından belirlenen teslimattan veya genel protokole (tcp_v4_rcv) teslimattan önce yapılır. Bu yapısal mimari nedeniyle, TCP yığınının paketi tüketip (consume edip) raw soketten gizlemesi söz konusu değildir.2g. Reverse Path Filtering (rp_filter) EtkisiLinux'ta asimetrik rotalamayı önleyen rp_filter mekanizması, paketin kaynak IP adresine ulaşılabilecek en iyi rotanın (FIB üzerinden) paketin geldiği arayüz olup olmadığını kontrol eder. Gönderilen hedef paket 1.1.1.1 üzerinden enp37s0 arayüzüne döndüğünde ve kaynak IP sahte olmadığında, ters yol doğrulama testini başarıyla geçer. rp_filter, hedef paketin düşürülmesinden sorumlu değildir.2h. Raw Socket Teslimat Zinciri (Protocol Demux)Bir IPv4 paketi ip_local_deliver_finish() fonksiyonuna ulaştığında, çekirdek paketi kopyalamadan önce raw_local_deliver() işlevini çağırır. Bu fonksiyon, IP başlığındaki protokol numarasını (TCP için 6) kullanarak raw_v4_htable tablosunda arama yapar. Eşleşen her SOCK_RAW soket için paketin sk_buff yapısı skb_clone() ile çoğaltılır ve ilgili soketin alım kuyruğuna gönderilir. Bu kural, giden SYN paketinin Linux bağlantı izleme (connection tracking) tablosuna (nf_conntrack) girilmesini engeller. Ancak, aynı port için gelen trafiğe INPUT yönünde simetrik bir NOTRACK uygulanmamıştır. Sunucudan dönen SYN-ACK paketi ağ yığınına girdiğinde, nf_conntrack_in modülü bu paketi herhangi bir başlatılmış oturumla eşleştiremez. Bir TCP bağlantısının yeni (NEW) bir oturum sayılabilmesi için pakette sadece SYN bayrağının bulunması gerekir; SYN-ACK bayrakları geçersizdir. Bu nedenle nf_conntrack, paketi INVALID olarak işaretler. Instruction-by-Instruction ÇözümlemeC0: {.code = 0x30,.jt = 0,.jf = 0,.k = 9 } // BPF_LD | BPF_B | BPF_ABS 
// Çekirdek İşlemi: IP Başlığının 9. ofsetindeki baytı Accumulator'a (A) yükle. 
// Gerçeklik: AF_INET raw soketlerinde offset 0 her zaman IP versiyon/IHL baytıdır. 9. ofset, L3 Transport Protokolüne eşittir (Örn: TCP için 6). Doğru bir adımdır.

1: {.code = 0x15,.jt = 0,.jf = 8,.k = 6 } // BPF_JMP | BPF_JEQ | BPF_K
// Çekirdek İşlemi: Eğer A == 6 (IPPROTO_TCP) ise bir sonraki satıra geç (jt=0). Değilse 8 satır atlayarak 10. satıra (DROP) git (jf=8). 

2: {.code = 0x20,.jt = 0,.jf = 0,.k = 16 } // BPF_LD | BPF_W | BPF_ABS
// Çekirdek İşlemi: IP Başlığının 16. ofsetindeki 32-bit veriyi (Destination IP) A'ya yükle.

3: {.code = 0x15,.jt = 0,.jf = 6,.k = src_ip } // BPF_JMP | BPF_JEQ | BPF_K
// Çekirdek İşlemi: A == src_ip ise devam et, değilse 6 satır atlayıp DROP'a git.
// KANIT AÇIĞI: Gelen paketteki Hedef IP adresi yerel cihazın kendi adresidir (192.168.1.2). Kuraldaki `src_ip` değişkeni Zig kodunda 1.1.1.1 olarak verildiyse filtre çöker. 

4: {.code = 0xb1,.jt = 0,.jf = 0,.k = 0 } // BPF_LDX | BPF_B | BPF_MSH
// Çekirdek İşlemi: Offset 0'daki baytı (IHL) al, ilk 4 bitini maskele, 4 ile çarp ve X (Index) register'ına yaz.
// Gerçeklik: Bu klasik BPF kuralı IPv4 başlığının gerçek uzunluğunu kusursuz hesaplar. Eğer IHL=5 ise, X=20 olur.

5: {.code = 0x48,.jt = 0,.jf = 0,.k = 0 } // BPF_LD | BPF_H | BPF_IND
// Çekirdek İşlemi: X(20) + 0 ofsetinden başlayarak 2 bayt oku (TCP Source Port). 
// KANIT: Okunan ağ (network) bayt değeri, BPF motoru tarafından otomatik olarak host (işlemci) bayt değerine çevrilip A'ya yazılır.

6: {.code = 0x15,.jt = 0,.jf = 3,.k = target_port } // BPF_JMP | BPF_JEQ | BPF_K
// Çekirdek İşlemi: A == target_port ise devam et, değilse DROP.
// KANIT: Dönen pakette kaynak port, sunucunun portudur (443).

7: {.code = 0x48,.jt = 0,.jf = 0,.k = 2 } // BPF_LD | BPF_H | BPF_IND
// Çekirdek İşlemi: X(20) + 2 ofsetinden başlayarak 2 bayt oku (TCP Destination Port).

8: {.code = 0x15,.jt = 0,.jf = 1,.k = ephemeral_port } // BPF_JMP | BPF_JEQ | BPF_K
// Çekirdek İşlemi: A == ephemeral_port ise devam et, değilse DROP.

9: {.code = 0x06,.jt = 0,.jf = 0,.k = 0x00040000 } // BPF_RET | BPF_K
// Çekirdek İşlemi: ACCEPT. Paketin tamamını kullanıcı alanına teslim et.

10: {.code = 0x06,.jt = 0,.jf = 0,.k = 0 } // BPF_RET | BPF_K
// Çekirdek İşlemi: DROP. Paketi yoksay.

TX/RX Aynı FD Mimarisi Analizi7a. Linux'ta Aynı FD (Soket) Mimarisi GüvenilirliğiLinux ağ mimarisinde, tek bir AF_INET, SOCK_RAW, IPPROTO_TCP dosya tanımlayıcısı (FD) üzerinden hem sendto() ile özel (crafted) TCP paketleri göndermek, hem de recvfrom() ile yanıtları okumak tamamen desteklenen, yasal ve güvenilir bir yaklaşımdır. Aynı soketin hem okuma hem yazma yönleri (duplex) raw operasyonlarda bağımsız kilit mekanizmalarına (lock) sahiptir.7b. Loopback (Yansıma) RiskiIP_MULTICAST_LOOP aktif olmadığında veya hedef IP adresi (1.1.1.1) yerel bir yönlendirme adresini temsil etmediğinde, Linux Kernel'ın IP yönlendirme algoritmaları (ip_route_output) giden unicast paketi doğrudan harici ağ kartına (NIC) iletir. Paket harici ağa kopyalandıktan sonra, yerel AF_INET kuyruğuna yansımaz (loopback yapmaz). Velev ki nadir bir hata nedeniyle paket kendi kuyruğuna düşse dahi; cBPF filtre bloğundaki Çapraz Port Karşılaştırması (target_port == 443 olmalı kuralı), giden SYN paketindeki kaynak port 62893 olduğu için paketi derhal imha (DROP) edecektir. Maskeleme riski sıfırdır.7c. Ayrı RX / TX FD AçmakAlma (RX) ve gönderme (TX) işlemleri için iki ayrı AF_INET/SOCK_RAW soketi açılması bu arızada hiçbir farklılık yaratmaz. Çünkü çekirdeğin paket teslimat fonksiyonu raw_local_deliver(), kayıtlı tüm IPPROTO_TCP soketlerine paketin birer kopyasını yönlendirir. Ancak, RX işlemi için AF_PACKET kullanılması, bölüm 10'da mimari karar olarak incelenmiştir.8. Kernel Delivery Path Analizi (Çekirdek Teslimat Yolu)Ağ kartı tarafından alınan SYN-ACK paketinin, hedeflenen recvfrom() fonksiyonuna ulaşamamasının en kritik çekirdek evresi burada izah edilmektedir.8a. ip_early_demux EtkisiLinux Çekirdeği 3.6+ sürümleri (ve mevcut 6.x sürümü) varsayılan olarak net.ipv4.ip_early_demux=1 değerine sahiptir. Bu özellik, paketin yönlendirme masraflarından kurtulması için erken soket araması (socket lookup) yapar. TCP yığını paketi tüketecek (consume) olsa dahi, net/ipv4/ip_input.c dosyasındaki ip_local_deliver_finish() süreci incelendiğinde; raw_local_deliver() fonksiyonunun TCP yığınından tamamen bağımsız olarak, yönlendirme (routing) aşamasından hemen sonra ve TCP/UDP teslimatından önce çağrıldığı görülür. ip_early_demux mekanizması, raw soketin paketi kopyalamasını engellemez.8b. conntrack / NOTRACK Etkileşimi (Birincil Kök Neden)Kanıtlarda, Zig kodunun işletim sistemi ağ katmanına gönderim anında aşağıdaki Netfilter/iptables kuralını enjekte ettiği belirlenmiştir:iptables -t raw -A OUTPUT -p tcp --sport 62893 -j NOTRACKBu konfigürasyon, Linux Çekirdeği'nin paket filtreleme mimarisindeki en agresif "Sessiz İmha" döngüsünü tetikler:Gönderilen SYN paketi Netfilter OUTPUT zincirine ulaşır. NOTRACK kuralı nedeniyle paket nf_conntrack bağlantı izleme tablosundan atılır (Bypass).Çekirdeğin bağlantı durumu haritasında SYN_SENT durumu oluşturulmaz.Karşı sunucudan (1.1.1.1) geri dönen SYN-ACK paketi L2 katmanını geçip ip_rcv içerisine girdiğinde, PREROUTING zincirinde nf_conntrack_in() fonksiyonu tarafından incelenir.Bağlantı tablosunda önceden kaydedilmiş bir oturum bulunamaz. Yeni bir oturum açılması (NEW state) için TCP standartlarına ve çekirdek izleme yapılarına (nf_conntrack_proto_tcp.c) göre gelen ilk paketin sadece SYN bayrağı taşıması zorunludur. Ancak gelen paket SYN-ACK bayrağı taşıdığı için çekirdek bu paketi hiçbir bağlantıyla ilişkilendiremez.Bunun sonucunda nf_conntrack, paketi INVALID (Geçersiz) durumuna atar.CachyOS, Arch Linux ve diğer modern dağıtımların varsayılan firewalld, ufw veya nftables kural setleri içerisinde şu kural mevcuttur:-A INPUT -m conntrack --ctstate INVALID -j DROPPaket Netfilter LOCAL_IN zincirinde sessizce imha edilir. Paket imha edildiği için ip_local_deliver_finish() fonksiyonu hiç çağrılmaz ve AF_INET raw soketi aç kalır.8c. rp_filter EtkisiSysctl parametrelerinden rp_filter (Reverse Path Filter), paketin geldiği arayüz (enp37s0) ile paketin kaynak adresine (1.1.1.1) ulaşmak için yönlendirme tablosunun işaret ettiği arayüzün uyuşup uyuşmadığını kontrol eder. Çıktılardaki rotalama simetrik olduğundan rp_filter testten başarıyla geçilir, bu değişken soruna dahil değildir.8d. tcpdump vs. Raw Socket Teslimat GarantisiBu analizdeki en sağlam dayanak tcpdump'ın paketi yakalayabilmesidir. tcpdump (libpcap), soket açarken AF_PACKET kullanır ve bu kanca (hook), paket L2 donanım seviyesinde dev_add_pack çağrısıyla ağ yığınına girer girmez çalışır. Paket daha sonra L3 IP katmanına, yönlendirme (routing) sistemlerine ve iptables/Netfilter (PREROUTING ve LOCAL_IN) engellerine yönlendirilir. AF_INET/SOCK_RAW soketleri ise paketi tüm bu cehennem zincirinden kurtulduktan sonra, ip_local_deliver_finish() fonksiyonunda teslim alır. Bu asimetri, paketin L2 (tcpdump) ve L3 (raw soket) arasında iptables tarafından düşürüldüğünün ispatıdır.9. Minimal Fix veya Mimari KararMevcut açmazı ortadan kaldırmak için sistem ve kod seviyesinde kademeli bir plan uygulanmalıdır.İlk Çözüm (Minimal Fix - Netfilter Asimetrisinin Düzeltilmesi)Sistemin gelen paketi güvenlik duvarından (conntrack) sağ salim geçirmesi için, INPUT/PREROUTING zincirine asimetriyi kıracak bir NOTRACK kuralı ilave edilmelidir.Çalıştırılacak tek satırlık sistem komutu:Bashsudo iptables -t raw -A PREROUTING -p tcp --sport 443 -d 192.168.1.2 -j NOTRACK
Açıklama: Bu kural, sunucudan dönen SYN-ACK paketinin de conntrack tablosu dışında (untracked) tutulmasını sağlar ve paketin INVALID bayrağı yiyip DROP kurallarına kurban gitmesini kesin olarak engeller.İkinci Çözüm (Kod Seviyesi - Endianness Optimizasyonu)Eğer güvenlik duvarı pasif olduğu halde sorun devam ediyorsa, cBPF (Zig) kodunda hata mevcuttur. Çekirdek içerisindeki BPF_LD | BPF_H talimatları ağdaki veriyi doğrudan CPU endian (Host Byte Order) mantığına dönüştürür.Zig kodundaki filter dizisi yapılandırılırken target_port ve ephemeral_port değişkenleri htons() eşdeğeri operasyonlara asla maruz bırakılmamalı, doğrudan yerel tamsayı (örn: 443 ve 62893) olarak k değerine atanmalıdır.İdeal Mimari Karar (AF_PACKET Kullanımı)C/Zig gibi diller ile asenkron ağ tarama (scanning) araçları geliştirilirken, dinleme (RX) soketi olarak AF_INET / SOCK_RAW kullanılması Linux ekosisteminde kırılgan bir mimari sayılır. Profesyonel tarayıcılar (örn: zmap, masscan) gelen paketleri tcpdump'ın yaptığı gibi L2 donanım seviyesinde okurlar.
Uygulanması Gereken Mimari Revizyon:Gönderim (TX): socket(AF_INET, SOCK_RAW, IPPROTO_TCP) (Mevcut haliyle bırakılır, işletim sistemi yönlendirmeyi yönetir).Dinleme (RX): socket(AF_PACKET, SOCK_RAW, htons(ETH_P_IP)) olarak açılmalıdır.
