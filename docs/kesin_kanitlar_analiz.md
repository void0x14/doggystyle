**1. Kesin kanıtlanan gerçekler**  
A-J maddeleri tam olarak kanıtlanmış durumdadır (verify.sh mesajı bilgilendirmedir, NOTRACK sadece OUTPUT sport scoped’tir, tcpdump SYN-ACK’ı görmektedir, program timeout olmaktadır, ilk SO_ATTACH_FILTER EFAULT idi ve düzeltildi, filter attach başarılı ancak SYN-ACK hâlâ yakalanmıyor, bind port scope etmiyor, arayüz enp37s0 doğru).

**2. Linux raw socket semantiği**  
AF_INET / SOCK_RAW / IPPROTO_TCP üzerinde recvfrom() buffer’ı IPv4 header ile başlar (L2/Ethernet header yoktur). cBPF filter (SO_ATTACH_FILTER) ile recvfrom() aynı veriyi görür. L2 header yoktur. bind(local_ip, 0) TCP port scope etmez (kanıt I’dan unrelated TCP/443 trafiği raw socket’e düşüyordu).

**3. Mevcut filter neden yanlış**  
Mevcut 24-instruction filter instruction-by-instruction incelendiğinde:  
- opcode’lar (0x30=ldb abs, 0x15=jeq, 0x54=and k, 0x64=lsh k, 0x07=tax, 0x50=ldb ind) ve jump zinciri (pc = current + 1 + jf) mantıken RET 0 (drop) konumuna (insn 23) doğru atlıyor.  
- IP dst offsetleri 16-19 ve TCP port byte’ları (x+0..3) doğru.  
- Ancak IHL hesabı (ldb[0] + and 0x0F + lsh #2 + tax) standart değildir ve bu kurulumda X register’ı SYN-ACK (options içeren paket) için doğru set edemiyor. Standart yol BPF_MSH (tek instruction, opcode 0xb1) ile X = 4*(pkt[0] & 0xf) kullanmaktır. Bu yüzden filter attach başarılı olsa da hedef SYN-ACK’i drop ediyor.

**4. Doğru exact classic cBPF**  
İstenen eşleşme (IPv4 dst == local IP, TCP src port == target 443, TCP dst port == ephemeral) için exact program (21 instruction, MSH + byte checks, jumps pc = current+1+jf semantiğiyle hesaplandı, drop insn 20’da):

```zig
const filter = [21]sock_filter{
    .{ .code = 0x30, .jt = 0, .jf = 0, .k = 9 },                  // 0: ldb [9] proto
    .{ .code = 0x15, .jt = 0, .jf = 18, .k = posix.IPPROTO.TCP }, // 1: jeq TCP
    .{ .code = 0x30, .jt = 0, .jf = 0, .k = 16 },                 // 2: ldb [16] dstIP[0]
    .{ .code = 0x15, .jt = 0, .jf = 16, .k = local_ip_bytes[0] },
    .{ .code = 0x30, .jt = 0, .jf = 0, .k = 17 },
    .{ .code = 0x15, .jt = 0, .jf = 14, .k = local_ip_bytes[1] },
    .{ .code = 0x30, .jt = 0, .jf = 0, .k = 18 },
    .{ .code = 0x15, .jt = 0, .jf = 12, .k = local_ip_bytes[2] },
    .{ .code = 0x30, .jt = 0, .jf = 0, .k = 19 },
    .{ .code = 0x15, .jt = 0, .jf = 10, .k = local_ip_bytes[3] },
    .{ .code = 0xb1, .jt = 0, .jf = 0, .k = 0 },                  // 10: ldxb 4*([0]&0xf)  (BPF_LDX|BPF_B|BPF_MSH)
    .{ .code = 0x50, .jt = 0, .jf = 0, .k = 0 },                  // 11: ldb [x+0] TCP src MSB
    .{ .code = 0x15, .jt = 0, .jf = 7, .k = target_port_bytes[0] },
    .{ .code = 0x50, .jt = 0, .jf = 0, .k = 1 },
    .{ .code = 0x15, .jt = 0, .jf = 5, .k = target_port_bytes[1] },
    .{ .code = 0x50, .jt = 0, .jf = 0, .k = 2 },                  // TCP dst MSB
    .{ .code = 0x15, .jt = 0, .jf = 3, .k = ephemeral_port_bytes[0] },
    .{ .code = 0x50, .jt = 0, .jf = 0, .k = 3 },
    .{ .code = 0x15, .jt = 0, .jf = 1, .k = ephemeral_port_bytes[1] },
    .{ .code = 0x06, .jt = 0, .jf = 0, .k = 0x00040000 },         // 19: RET accept (full packet)
    .{ .code = 0x06, .jt = 0, .jf = 0, .k = 0 },                  // 20: RET drop
};
```

- IHL değişken: BPF_MSH (0xb1) ile exact ele alınır (X register doğrudan set edilir).
- Endian: port ve IP byte’ları packet network order (BE) ile eşleşir.
- Accept/drop akışı: tüm match → insn 19 (accept), herhangi mismatch → RET 0 (drop).

**5. Exact C ABI**  
```c
struct sock_filter {
    __u16 code;   /* Actual filter code */
    __u8  jt;     /* Jump true */
    __u8  jf;     /* Jump false */
    __u32 k;      /* Generic multiuse field */
};

struct sock_fprog {
    unsigned short len;
    struct sock_filter __user *filter;
};
```

**6. Exact Zig ABI**  
En güvenli:
```zig
const sock_filter = extern struct {
    code: u16,
    jt: u8,
    jf: u8,
    k: u32,
};
const sock_fprog = extern struct {
    len: u16,
    filter: [*c]const sock_filter,
};
```
- `extern struct` zorunlu: C ABI/layout ile birebir uyum (alignment, padding yok).  
- `packed struct` yanlış: padding/alignment bozar, setsockopt yanlış struct alır.  
- İlk EFAULT: muhtemelen non-extern struct veya .filter pointer geçersiz (bad address).

SYN-ACK paketinin yerel enp37s0 arayüzüne (interface) hatasız bir şekilde ulaştığının çekirdek seviyesinde tcpdump (ki o da AF_PACKET kullanır) tarafından tespit edilmesi, ağ yolunun, aradaki yönlendiricilerin, NAT (Ağ Adresi Çevirisi) cihazlarının veya MTU (Maksimum İletim Birimi) kısıtlamalarının bu senaryoda tamamen kusursuz çalıştığını ispatlar. Daha sonraki sürümde elde edilen setsockopt(..., {len=24, filter=0x7ffe5da7a840}, 16) = 0 çıktısı ise, boyut ve pointer hatalarının kısmen aşıldığını, çekirdeğin sunulan bayt dizisini geçerli bir BPF programı olarak kabul edip sokete bağladığını ispatlamaktadır.Son kanıt ise, filtre bağlamanın başarılı olmasına rağmen uygulamanın hala paketleri kaçırarak Handshake TIMEOUT hatası vermesidir. strace kayıtları, uygulamanın sendto çağrısı ile paketi gönderdikten sonra sürekli olarak recvfrom çağrısı yaptığını ve -1 EAGAIN (Resource temporarily unavailable) yanıtı aldığını göstermektedir. Non-blocking (bloklamayan) veya zaman aşımı (timeout) ayarlı bir sokette EAGAIN dönmesi, soketin alım kuyruğunun (receive queue) tamamen boş olduğu anlamına gelir. Paket ağ arayüzüne ulaşmış, ancak soketin kuyruğuna girmemiştir. Gelen paket, BPF programı tarafından değerlendirilir. Eğer BPF programı 0 döndürürse, paket sessizce reddedilir (drop) ve recvfrom çağrısına asla ulaşmaz. Bu kesin kanıtlar, fiziksel bir ağ hatası olmadığını, sorunun tamamen mevcut Classic BPF (cBPF) programının mantıksal yanlışlıklarından ve raw soket okuma semantiğine uygun olmayan offset (sapma) varsayımlarından kaynaklandığını şüpheye yer bırakmayacak şekilde doğrulamaktadır.2. Linux Raw Socket SemantiğiLinux işletim sisteminde ağ programlaması yapılırken, soketin yapılandırılma biçimi çekirdeğin paketleri nasıl işleyeceğini ve kullanıcı alanına nasıl sunacağını doğrudan belirler. Görev kapsamında kullanılan socket(AF_INET, SOCK_RAW, IPPROTO_TCP) sistem çağrısı, Linux IPv4 ağ yığınında son derece spesifik kurallara tabi olan bir iletişim kanalı yaratır. Bu semantiğin yanlış anlaşıldığı durumlarda filtreleme mekanizmaları kaçınılmaz olarak çöker.SO_ATTACH_FILTER ve Buffer Başlangıç Noktası (Offset 0)AF_INET adres ailesi ve SOCK_RAW tipi kullanılarak oluşturulan bir soket, çekirdeğin IP katmanı (Layer 3) üzerinden doğrudan iletişim kurmasını sağlar. Bu soket türünün en temel semantik kuralı, işletim sisteminin bağlantı katmanı (Layer 2 - Ethernet, MAC, 802.11 vb.) başlıklarını ağ arabirim kartı (NIC) sürücüleri seviyesinde sıyırmasıdır (strip).Bir ağ paketi kablodan NIC'e ulaştığında, çekirdek eth_type_trans() fonksiyonu aracılığıyla Ethernet başlığını okur, protokol tipini (EtherType) belirler ve bu başlığı bellek alanından (sk_buff->data) atlayarak paketi üst katmanlara iletir. Bu nedenle, AF_INET raw soketi üzerinde SO_ATTACH_FILTER ile bağlanan bir cBPF programı çalışmaya başladığında, sanal makinenin gördüğü veri arabelleği (buffer) doğrudan IPv4 başlığının (IP Header) ilk baytı ile başlar. Arabellekte kesinlikle hiçbir L2 (Ethernet) başlığı bulunmaz.Birçok ağ mühendisinin sıklıkla düştüğü hata, libpcap tabanlı araçların (örneğin standart tcpdump) veya AF_PACKET adres ailesinin davranışını AF_INET soketlerine uygulamaya çalışmaktır. AF_PACKET soketleri tüm L2 çerçevesini yakalar, bu nedenle paket offset 0'da MAC adresleri ile başlar ve IP başlığına ulaşmak için 14 bayt (Ethernet başlık uzunluğu) atlamak gerekir. Ancak AF_INET raw soketleri için IP başlığı tam olarak offset 0'dan başlar. BPF komutlarında IP protokolünü okumak için offset 23 (14+9) yerine offset 9 kullanılmalıdır. Mevcut kanıtlar, Zig uygulamasının bu ayrımı doğru yaptığına işaret etse de, sonraki TCP ofsetlerinde ve port okumalarında bu yapının derinleşen kuralları ihlal edilmiştir.recvfrom() ve Filtre GörünürlüğüÇekirdek mimarisinde paket alma süreci ip_rcv -> ip_local_deliver -> raw soket işleyicileri şeklinde ilerler. Bir paket sokete teslim edilmeden hemen önce sk_filter (veya modern çekirdeklerde sk_filter_trim_cap) çağrılır. Bu aşamada, BPF filtresinin eriştiği sk_buff (socket buffer) veri yükü, filtreden geçip kullanıcı alanındaki recvfrom() fonksiyonuna kopyalanacak olan veri yükü ile bayt bayt tamamen aynıdır.Yani, BPF programı offset 0'da IPv4 Versiyon/IHL (Internet Header Length) baytını görüyorsa, recvfrom() fonksiyonunun doldurduğu arabelleğin 0. baytı da tam olarak aynı IPv4 Versiyon/IHL baytıdır. Filtre, paketi analiz ettikten sonra sıfırdan farklı bir değer döndürürse (accept), paket o haliyle soket alım kuyruğuna (receive queue) eklenir. Eğer sıfır döndürürse (drop), paket kuyruğa hiç girmez. Bu, BPF'in ve recvfrom()'un paketi aynı ardışık (linear) düzende gördüğünü ispatlar.bind(local_ip, 0) ve TCP Port Scope (Kapsam) DavranışıGeçmiş kanıtlarda (I. Tarihsel Kanıt) gözlemlenen, ilgisiz HTTPS trafiğinin raw sokete düşmesi durumu, Linux IPv4 yığınının yapısal bir özelliğidir. AF_INET ve SOCK_RAW kullanılarak açılan bir sokette, bind() sistem çağrısı yalnızca L3 (Ağ Katmanı) yönlendirmesini etkiler. Yani bind(192.168.1.2, 0) çağrısı, soketi yalnızca belirli bir yerel IP adresine kilitler.Ancak, soket tipi SOCK_RAW olduğu için çekirdek bu soketi bir Taşıma Katmanı (Transport Layer - Layer 4) demultiplexer'ı (çoğullama çözücü) olarak değerlendirmez. Çekirdeğin kendi TCP yığını (TCP stack) bağlantı bazlı 4-tuple (kaynak ip, hedef ip, kaynak port, hedef port) eşleştirmesi yaparken, IPPROTO_TCP parametresi ile oluşturulmuş bir raw soket, o yerel IP'ye gelen tüm TCP paketlerinin bir kopyasını ("clone") alır. Çekirdek, raw soketin port numaralarını anlayabildiğini varsaymaz; tüm yükü kullanıcı uygulamasına bırakır. Bu nedenle, aynı makinedeki Apache veya Nginx sunucusuna gelen TCP 443 trafiği, TCP yığınına iletilirken eşzamanlı olarak raw sokete de yönlendirilir.Eğer BPF filtresi kullanılmazsa, raw soket bu yüksek hacimli ilgisiz trafikle dolar. recvfrom() hızla EAGAIN yerine ilgisiz paketler döndürmeye başlar. Uygulamanın kendi bağlamına (context) uygun olan spesifik SYN-ACK paketini bulması için binlerce paketi kullanıcı alanına kopyalayıp tek tek okuması gerekir ki bu süreç darboğaza (bottleneck) ve neticesinde kanıtlarda görülen zaman aşımı (timeout) hatalarına yol açar. Çekirdek tarafında port bazlı ayıklamanın yapılabilmesinin tek yolu, doğru tasarlanmış bir cBPF programını SO_ATTACH_FILTER ile sokete yerleştirmektir.3. Zig kodunda bu sayı üzerinden src_ip >> 24 işlemi yapıldığında, işlemci en anlamlı (most significant) bayt olan 0x02 (yani 2) değerini alır ve local_ip_bytes içine 2 yazar.BPF sanal makinesi paketten offset 16'daki ilk baytı okuduğunda (LD_B_ABS 16), gerçek ağ paketi olan 192 (0xC0) değerini Akümülatöre yükler. Filtre, 192 ile 2'yi karşılaştırır. Eşleşme başarısız olur ve BPF komutu jf (Jump False) rotasına, yani paket reddetme (DROP) fonksiyonuna dallanır. Filtre, ağ bayt sırası ile işlemci (host) bayt sırasını manuel olarak ve hatalı bir şekilde birbirine karıştırmıştır.Optimizasyon Eksikliği ve Gereksiz KompleksiteIP adresini ve 16-bitlik port numaralarını bayt bayt okuyup 4 ayrı komutta eşleştirmek (toplam 12 komut), Linux BPF mimarisi için anti-pattern (kötü tasarım) kabul edilir. BPF mimarisinde BPF_W (32-bit Word) ve BPF_H (16-bit Half-word) yükleme komutları mevcuttur. Çekirdek, bu komutları yürüttüğünde ağ paketindeki veriyi okur ve işlemcinin çalışabileceği Host Byte Order formatına otomatik olarak dönüştürür (ntohl ve ntohs işlemleri kernel içerisinde JIT derleyici tarafından şeffaf olarak uygulanır). Dolayısıyla tek bir komutla tüm IP adresi veya tek bir komutla tüm Port numarası kontrol edilebilecekken, manuel bayt ayrıştırma işlemi hem hata payını artırmış hem de jump (atlama) offsetlerini yönetilemez hale getirmiştir.4. Exact C ABILinux çekirdeği ile iletişimde veri yapıları bellek düzenlerine (memory layout) son derece duyarlıdır. 64-bit bir Linux işletim sisteminde linux/filter.h dosyası üzerinden içe aktarılan C yapılarının tam boyutları ve bellek hizalamaları (alignment) şöyledir :Cstruct sock_filter { // 8 Byte Toplam Boyut
    __u16 code;      // Offset 0 (2 byte)
    __u8  jt;        // Offset 2 (1 byte)
    __u8  jf;        // Offset 3 (1 byte)
    __u32 k;         // Offset 4 (4 byte)
};

struct sock_fprog {  // 16 Byte Toplam Boyut
    unsigned short len;                // Offset 0 (2 byte)
    // --- 6 BYTE COMPILER PADDING --- // Offset 2-7 arası dolgu baytları
    struct sock_filter __user *filter; // Offset 8 (8 byte pointer - 64-bit)
};
Çekirdekte setsockopt sistem çağrısı işlenirken sock_fprog yapısı, copy_from_user aracılığıyla kullanıcı alanından kopyalanır. 64-bit C ABI (Application Binary Interface) kuralları gereği, işaretçiler (pointers) 8 baytlık bellek adreslerine hizalanmak zorundadır (8-byte boundary alignment). len değişkeni yalnızca 2 bayt yer kapladığı için, C derleyicisi araya 6 baytlık görünmez bir dolgu (padding) yerleştirerek işaretçinin belleğin 8. baytından başlamasını garanti eder. Bu padding eksik olduğunda tüm bellek haritası kayar.6. Exact Zig ABIZig programlama dili varsayılan olarak veri yapılarının (structs) bellek düzenini performans veya güvenlik gerekçesiyle otonom bir şekilde değiştirebilir. C ABI ile uyumluluğun mutlak olduğu sistem çağrıları (syscall) sınırında bu durum tehlikelidir.Güvenli Zig EşdeğeriAşağıdaki kod parçası, C çekirdek beklentilerini birebir yansıtan en güvenli Zig eşdeğeridir:Code snippetconst std = @import("std");

pub const sock_filter = extern struct {
    code: u16,
    jt: u8,
    jf: u8,
    k: u32,
};

pub const sock_fprog = extern struct {
    len: u16,
    filter: [*]const sock_filter,
};
Neden extern struct?extern struct anahtar kelimesi, Zig derleyicisine bu veri yapısının hedef platformun (x86_64 Linux) geçerli C ABI kurallarına uygun olarak derlenmesini zorunlu kılar. Bu bildirim kullanıldığında Zig, 64-bit işaretçinin 8 bayt hizalama kuralına uyması gerektiğini bilir ve len alanının arkasına otomatik olarak 6 baytlık bir padding yerleştirir. Bu sayede sock_fprog'un toplam boyutu @sizeOf(sock_fprog) == 16 olur.Neden packed struct Yanlış ve EFAULT'un Nedenipacked struct, yapının içerisindeki tüm dolgu (padding) baytlarını tamamen ortadan kaldırarak verileri peş peşe sıkıştırır. Eğer sock_fprog packed olarak tanımlanmış olsaydı, filter işaretçisi 2. bayttan başlardı ve yapının toplam boyutu 10 bayt olurdu.İlk denemede alınan setsockopt(..., 16) = -1 EFAULT hatasının mutlak nedeni budur. Geliştirici çekirdeğe len=16 baytlık bir alan okumasını söyler, ancak Zig bellek modeli packed ise, çekirdek 8. bayttan itibaren filter işaretçisini okumaya kalktığında, bellekteki o konum pointer değildir; yapı bitmiştir ve bitişik bellek çöpleri veya korumalı yığın (stack) alanları okunmuştur. Çekirdek bu geçersiz veya eşleştirilmemiş (unmapped) adresten filtre dizisini kopyalamaya çalıştığında, bir Donanım Sayfa Hatası (Hardware Page Fault) tetiklenir ve işlem -1 EFAULT koduyla kullanıcıya geri fırlatılır.Pointer Alanı TipiPointer alanı [*]const sock_filter (many-item pointer) olmalıdır. Zig'de *sock_filter (single-item pointer) sadece tek bir öğeyi işaret ederken, [*]const belirsiz sayıda öğeden oluşan bir C dizisine işaret eder. Bu, C'deki struct sock_filter * karşılığına hem anlamsal hem de boyutsal olarak eşdeğerdir.7.
